# 文件目的

本文件定義任務態系統相關說明文件的分層等級。

透過等級分類，使不同文件具備明確責任範圍，避免核心概念、規則、流程與實作內容混合。

文件等級由高至低依序分為：

Level 0
系統定位層

Level 1
規則結構層

Level 2
流程交互層

Level 3
實作驗證層


# Level 0

## 系統定位層

負責：

核心概念

整體架構

設計邊界


說明：

Level 0 用於定義系統存在目的與基本定位。

主要回答：

為什麼需要此系統。

此系統解決什麼問題。

此系統與其他系統的差異。

此系統的範圍與限制。


不可包含：

詳細規則。

資料欄位。

程式實作。


# Level 1

## 規則結構層

負責：

判定規則

資料結構

狀態定義


說明：

Level 1 用於建立系統運作所需要的基本結構。

主要回答：

何時建立狀態。

狀態包含什麼。

資料如何組織。

規則如何判定。


不可包含：

完整執行流程。

實際工程部署。


# Level 2

## 流程交互層

負責：

執行流程

模組交互

資料流向


說明：

Level 2 用於描述系統如何運作。

主要回答：

資料如何流動。

模組如何互動。

任務如何從開始到完成。


不可包含：

底層程式細節。

特定技術實作。


# Level 3

## 實作驗證層

負責：

工程規格

測試驗證

效益評估


說明：

Level 3 用於將架構轉換為可執行與可驗證內容。

主要回答：

如何建立。

如何測試。

如何確認有效。


包含：

開發規格。

測試方法。

效能觀察。

實際應用驗證。


# 等級關係

Level 0

決定系統方向。


↓

Level 1

建立系統規則與結構。


↓

Level 2

定義系統運作流程。


↓

Level 3

進行實作與驗證。


# 文件建立原則

文件應依照等級限制內容範圍。

高等級文件不應直接包含低等級細節。

低等級文件不得修改高等級核心定位。


目的：

維持任務態系統文件的一致性。

降低架構混亂。

支援後續擴充與維護。
